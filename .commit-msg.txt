fix: no_profile redirect loop + email-based user fallback

Problem: A user who signs in via Google but whose auth.users.id does not
match a public.users row (because seed.sql used a wrong/placeholder UUID,
or because they signed up via Google before being seeded) got stuck in
an infinite redirect loop:

  /login -> proxy sees user is authed -> /overview?error=no_profile
  /overview -> getCurrentUser sees no public.users -> /login?error=no_profile
  /login -> proxy sees user is authed, query params preserved by clone() -> ...

Fixes:

- proxy.ts: when bouncing an authed user from /login to /overview,
  strip query params (url.search = ""). Also: when the URL carries
  error=no_profile, DO NOT bounce — let /login render so the user
  sees what is wrong.

- getCurrentUser: when public.users.id lookup fails, fall back to
  email lookup via the service-role admin client (bypassing RLS).
  This auto-links a seeded public.users row whose id does not match
  the auth.users.id, as long as the email matches.

- login/page.tsx: when on /login?error=no_profile and a user is signed
  in, render a diagnostic panel showing the auth user email + id with
  a copy-paste SQL snippet to update public.users.id, plus a sign-out
  button. The login form itself is hidden in this state (you cannot
  log in over an existing session anyway).

Result: no more loop. The user sees a clear error with a one-line
recovery SQL or a sign-out button.
